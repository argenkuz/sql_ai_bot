# SQL Design: Views, Procedures, Indexes

Этот файл объясняет, как в проекте Spotify Data Studio Bot используются SQL views, stored procedures и indexes, зачем они нужны и какую бизнес-проблему помогают решить.

## Business Problem

Музыкальная база содержит треки, артистов, альбомы, жанры и аудио-характеристики. Без аналитического слоя пользователю сложно быстро понять:

- какие треки и жанры самые сильные;
- какие артисты чаще всего встречаются;
- какие музыкальные характеристики влияют на популярность;
- как быстро получать отчёты без ручного написания SQL.

Бизнес-проблема проекта: превратить сырые Spotify-данные в быстрый и понятный аналитический инструмент, который помогает принимать решения по плейлистам, рекомендациям, контент-стратегии и музыкальной аналитике прямо через Telegram.

## Problem Statements

### Problem 1: Трудно быстро находить самые популярные треки

Пользователю нужно быстро получить топ треков по популярности. Если каждый раз писать полный SQL вручную, это занимает время и повышает риск ошибок.

Решение в проекте:

- view `top_tracks`;
- procedure `get_top_tracks(limit_count)`;
- бот-команда `/top_tracks`;
- AI-запросы обычным текстом вроде "скинь топ 10 песен".

Это упрощает получение рейтингов и делает аналитику доступной без знания SQL.

### Problem 2: Непонятно, какие жанры лучше работают

Для музыкальной аналитики важно понимать, какие жанры имеют большую среднюю популярность. Без готовой агрегации нужно каждый раз соединять `tracks` и `genres`, группировать данные и считать `AVG(popularity)`.

Решение в проекте:

- view `genre_popularity`;
- procedure `genre_stats(genre_name_input)`;
- бот-команды `/genre_map`, `/mood`, `/energy`;
- AI-вопросы вроде "какие жанры самые энергичные".

Это помогает сравнивать жанры и находить направления, которые потенциально лучше заходят аудитории.

### Problem 3: Нужно анализировать альбомы по длительности

Для каталогизации музыки и подготовки подборок полезно понимать суммарную длительность альбомов. Без view приходится каждый раз соединять `tracks` и `albums`, затем считать сумму `duration_ms`.

Решение в проекте:

- view `album_duration`;
- возможность быстро расширить бота командой для анализа альбомов;
- AI может использовать похожую логику для вопросов про длинные треки и альбомы.

Это помогает находить длинные альбомы, короткие релизы и планировать плейлисты по времени.

### Problem 4: Запросы с JOIN могут работать медленно

Бот часто соединяет таблицы:

- `tracks` с `genres`;
- `tracks` с `albums`;
- `tracks` с `audio_features`;
- `tracks` с `track_artist`;
- `track_artist` с `artists`.

Если нет индексов по ключам связей, такие запросы могут тормозить при росте базы.

Решение в проекте:

- `idx_tracks_album`;
- `idx_tracks_genre`;
- `idx_track_artist_track`;
- `idx_track_artist_artist`;
- `idx_audio_track`.

Эти индексы помогают MySQL быстрее находить связанные строки и ускоряют отчёты, графики и AI-generated SQL.

### Problem 5: Пользователь не должен писать SQL руками

Даже если база хорошо нормализована, обычному пользователю неудобно помнить названия таблиц и колонок.

Решение в проекте:

- Telegram-бот принимает вопросы обычным текстом;
- OpenRouter превращает вопрос в безопасный `SELECT`;
- views и procedures дают готовые аналитические шаблоны;
- indexes ускоряют выполнение запросов.

В итоге пользователь спрашивает: "покажи самые танцевальные треки", а бот сам строит запрос, получает данные и красиво оформляет ответ.

## Views

Views нужны, чтобы спрятать повторяющуюся SQL-логику за понятным названием. Они делают проект чище: вместо длинного запроса можно обращаться к готовому представлению.

### `top_tracks`

```sql
CREATE VIEW top_tracks AS
SELECT
    track_name,
    popularity,
    duration_ms
FROM tracks
ORDER BY popularity DESC;
```

Что решает:

- быстро показывает треки по популярности;
- упрощает top-ranking аналитику;
- может использоваться для отчётов, где не нужны артисты и жанры.

Как помогает боту:

- логика "популярные треки" уже вынесена на уровень базы;
- похожая логика используется в `/top_tracks` и AI-запросах про топ песен;
- можно быстро добавить отдельную команду, которая читает прямо из `top_tracks`.

### `genre_popularity`

```sql
CREATE VIEW genre_popularity AS
SELECT
    g.genre_name,
    AVG(t.popularity) AS avg_popularity
FROM tracks t
JOIN genres g ON t.genre_id = g.genre_id
GROUP BY g.genre_name;
```

Что решает:

- считает среднюю популярность по жанрам;
- убирает необходимость каждый раз писать `JOIN + GROUP BY`;
- помогает сравнивать жанры между собой.

Как помогает боту:

- поддерживает отчёты про жанры;
- даёт основу для вопросов "какой жанр популярнее";
- делает будущие genre-отчёты короче и понятнее.

### `album_duration`

```sql
CREATE VIEW album_duration AS
SELECT
    a.album_name,
    SUM(t.duration_ms) AS total_duration
FROM tracks t
JOIN albums a ON t.album_id = a.album_id
GROUP BY a.album_name;
```

Что решает:

- считает суммарную длительность альбомов;
- помогает анализировать релизы по времени;
- подходит для вопросов про длинные и короткие альбомы.

Как помогает боту:

- можно быстро добавить команду `/album_duration`;
- AI-запросы про длительность могут использовать эту идею;
- уменьшается дублирование SQL-логики.

## Stored Procedures

Procedures нужны для повторяемых операций, где заранее известна бизнес-логика и меняется только параметр.

Важно: в основном AI-режиме бот запрещает `CALL`, потому что это безопаснее для генерации SQL. Но процедуры всё равно полезны как готовый backend-слой для ручной проверки, будущих команд и демонстрации database logic.

### `get_top_tracks(limit_count)`

```sql
CREATE PROCEDURE get_top_tracks(IN limit_count INT)
BEGIN
    SELECT track_name, popularity
    FROM tracks
    ORDER BY popularity DESC
    LIMIT limit_count;
END
```

Что решает:

- возвращает нужное количество популярных треков;
- параметр `limit_count` позволяет получать топ 5, топ 10, топ 20;
- логика рейтинга хранится внутри базы.

Как помогает проекту:

- можно использовать для отдельной команды без генерации AI-SQL;
- удобно тестировать топ треков напрямую в MySQL;
- уменьшает повторение одинакового SQL.

Пример ручного вызова:

```sql
CALL get_top_tracks(10);
```

### `genre_stats(genre_name_input)`

```sql
CREATE PROCEDURE genre_stats(IN genre_name_input VARCHAR(100))
BEGIN
    SELECT
        g.genre_name,
        COUNT(t.track_id) AS total_tracks,
        AVG(t.popularity) AS avg_popularity
    FROM tracks t
    JOIN genres g ON t.genre_id = g.genre_id
    WHERE g.genre_name = genre_name_input
    GROUP BY g.genre_name;
END
```

Что решает:

- показывает статистику по конкретному жанру;
- считает количество треков;
- считает среднюю популярность.

Как помогает проекту:

- подходит для точечных вопросов по одному жанру;
- может быть основой для команды вроде `/genre_stats pop`;
- помогает быстро проверять качество данных по жанрам.

Пример ручного вызова:

```sql
CALL genre_stats('pop');
```

## Indexes

Indexes помогают базе быстрее выполнять `JOIN`, `WHERE`, `GROUP BY` и отчёты. В этом проекте они особенно важны, потому что данные нормализованы: информация о треке, артистах, жанре, альбоме и аудио-фичах находится в разных таблицах.

### `idx_tracks_album`

```sql
CREATE INDEX idx_tracks_album ON tracks(album_id);
```

Помогает быстрее соединять `tracks` и `albums`.

Используется в логике:

- анализ альбомов;
- view `album_duration`;
- вопросы про альбомы и длительность.

### `idx_tracks_genre`

```sql
CREATE INDEX idx_tracks_genre ON tracks(genre_id);
```

Помогает быстрее соединять `tracks` и `genres`.

Используется в логике:

- `/genre_map`;
- `genre_popularity`;
- `genre_stats`;
- вопросы про жанры.

### `idx_track_artist_track`

```sql
CREATE INDEX idx_track_artist_track ON track_artist(track_id);
```

Помогает быстрее находить артистов для конкретного трека.

Используется в логике:

- `/top_tracks`;
- `/dance`;
- `/energy`;
- `/tempo`;
- `/hidden_gems`;
- AI-запросы, где нужно показать трек вместе с артистами.

### `idx_track_artist_artist`

```sql
CREATE INDEX idx_track_artist_artist ON track_artist(artist_id);
```

Помогает быстрее находить все треки конкретного артиста.

Используется в логике:

- `/top_artists`;
- вопросы про самых частых артистов;
- подсчёт количества треков по артистам.

### `idx_audio_track`

```sql
CREATE INDEX idx_audio_track ON audio_features(track_id);
```

Помогает быстрее соединять `tracks` и `audio_features`.

Используется в логике:

- `/dance`;
- `/energy`;
- `/mood`;
- `/tempo`;
- `/audio_profile`;
- `/hidden_gems`;
- вопросы про danceability, energy, valence, tempo, acousticness.

## How These SQL Objects Simplify the Bot

Views simplify repeated analytics:

- вместо длинных запросов можно использовать готовые представления;
- бизнес-логика становится понятнее;
- будущие команды добавляются быстрее.

Procedures simplify parameterized reports:

- топ треков можно получать с разным лимитом;
- статистику жанра можно получать по входному параметру;
- часть аналитики можно держать прямо в базе.

Indexes improve performance:

- ускоряют частые joins;
- помогают боту быстрее отвечать;
- снижают нагрузку на MySQL;
- особенно полезны, когда датасет становится больше.

## Connection to the Telegram Bot

В `app.py` бот использует ту же аналитическую идею:

- готовые команды похожи на database views и procedures;
- AI-generated SQL строится только по разрешённой схеме;
- индексы автоматически используются MySQL при выполнении запросов;
- пользователь получает не SQL-таблицу, а понятный ответ, CSV или график.

Даже когда бот пишет SQL напрямую к базовым таблицам, созданные индексы всё равно помогают, потому что MySQL использует их при `JOIN` по `album_id`, `genre_id`, `track_id`, `artist_id`.

## Business Value

Эта SQL-структура делает проект полезным не только технически, но и бизнесово:

- быстрее находить популярный контент;
- понимать сильные жанры;
- анализировать артистов и треки;
- строить подборки по настроению и энергии;
- автоматизировать отчёты;
- давать пользователю аналитику без ручного SQL.

Итог: база превращается из набора таблиц в аналитический инструмент для музыкальных решений.
